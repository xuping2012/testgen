# testcase-generator 子Agent体系

## 概述

testcase-generator技能采用Multi-Agent串行Pipeline架构，将测试用例生成流程拆分为4个独立子Agent串行执行。每个Agent的输出作为下一个Agent的输入，确保各阶段职责清晰、便于问题追溯。

## Pipeline架构

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  需求分析    | →  │  模块评审    │ →  │  用例生成   | →  │  用例评审    │
│   Agent     │    │   Agent     │    │   Agent     │    │   Agent     │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
      ↓                  ↓                  ↓                  ↓
   功能模块            评审结论           测试用例           评审报告
   业务流程            通过/不通过         具体用例          导出脚本
   约束条件                             优先级分布
   测试点                                 用例统计
   测试策略
```

## 子Agent清单

| 序号 | Agent名称 | 职责 | 输入 | 输出 |
|------|-----------|------|------|------|
| 01 | 需求分析Agent | 读取需求文档，识别功能模块、业务流程、约束条件，并组织测试点 | 需求文档（PRD/MD/TXT/Word/PDF） | 功能模块清单、业务流程步骤、约束条件、测试点清单、映射表 |
| 02 | 模块评审Agent | 评审测试点划分是否合理，输出评审结论 | 需求分析Agent的测试点输出 | 评审结论（通过/不通过）+ 不通过原因 |
| 03 | 用例生成Agent | 基于测试点生成具体测试用例 | 模块评审Agent通过的测试点 | 测试用例（按模块+测试点组织）、用例统计 |
| 04 | 用例评审Agent | 执行质量评审，输出评审报告 | 用例生成Agent的输出 | 质量评审报告、重复检测结果、评审结论 |

## 执行流程

### 阶段1：评估方法依据（标准化引用）

- **需求文档**：PRD/MD/TXT/Word/PDF
- **测试策略**：功能测试、性能测试、安全测试、兼容性测试、接口测试、UI测试、回归测试
- **设计方法**：等价类、边界值、因果图、场景法、状态迁移、组合法、错误推测法、探索性测试
- **测试用例格式**：表格、思维导图、文档、脚本

### 阶段2：输入确认与模式识别

- **生成模式**：用户要求"生成测试用例"，按阶段1-7完整执行
- **评审模式**：用户要求"评审已有用例"，跳过阶段3-6，直接执行阶段7的质量评审
- **补充模式**：用户要求"补充测试用例"，先读取已有用例，对比需求文档识别未覆盖的场景，再执行阶段1-7

### 阶段3：需求解析（需求分析Agent）

1. 读取需求文档：识别功能模块清单、业务流程步骤、状态变化、角色(终端)
2. 提取业务流程：找出"第一步→第二步→...→结束"的操作序列
3. 识别约束：必填/可选、长度/格式/范围、错误码、权限规则
4. 划分测试点：按需求中的实际子功能/操作划分
5. 确定测试策略：选择测试类型和设计方法
6. 规划测试场景：正向、反向、异常、边界值
7. 输出：功能模块+业务流程+约束+测试点+映射表

### 阶段4：模块评审（模块评审Agent）

```
1. 需求分析Agent输出测试点清单
2. 模块评审Agent按4个维度进行评审
   - 覆盖完整度：是否覆盖PRD所有功能点
   - 结果合理性：模块划分是否高内聚低耦合
   - 层级清晰度：父子关系是否明确
   - 命名规范性：命名是否语义清晰
3. 评审结果：
   - 全部通过 → 进入阶段6（用例生成）
   - 任一维度不通过 → 返回阶段3（需求解析）重新生成
   - 重新生成 → 重新评审（循环直到通过）
```

### 阶段5：测试策略

1. 选择测试类型：默认功能测试，依据需求特征增加其他类型
2. 选择设计方法：根据约束条件选择等价类、边界值、因果图等方法
3. 场景映射：将约束映射到正向/反向/边界/异常场景
4. 输出：测试策略（类型+方法+场景清单）

### 阶段6：用例生成（用例生成Agent）

1. 正向场景（Happy Path）→ P0或P1
2. 边界值（min-1/min/max+1）→ P2
3. 异常场景（需求明确提及的异常分支）→ P2
4. 反向场景（需求明确提及的反向流程）→ P2或P3
5. 合并去重

### 阶段7：优先级校验与质量评审（用例评审Agent）

1. 优先级校验：统计分布、超标检查、缺失检查、P0审查
2. 质量评审：引导错误过滤 + 六大维度评估
3. 输出：质量评审报告
