## 思考链（CoT）

### 1. 业务流程识别

生成用例前，必须先识别业务功能流程：

#### 1.1 识别步骤

1. **寻找流程关键词**：
   - 动词：创建、分配、登录、签到、拆卸、提交、验收、驳回、点击、保存、取消、确认、重试、关闭、修改、删除、查看、搜索、导出、导入、配置、设置、返回等
   - 顺序词：步骤1、步骤2、首先、然后、接着、最后、第一步····第N步、步骤N····第N步等
   - 状态词：待分配、已分配、作业中、待验收、已关闭、在线、离线、待激活、待处理、待审核、待确认、已完成、已取消、进行中、上架、下架等
2. **识别流程参与者**：
   - Web端
   - app移动端
   - 微信/支付小程序端
3. **识别流程闭环**：
   - 正常流程：步骤1 → 步骤2 → ... → 步骤N → 结束
   - 异常流程：驳回、重试、返回等
   - 状态变化：每个步骤的状态变化

#### 1.2 输出流程

```
业务流程识别结果：
步骤1: [操作描述] → [状态变化]
步骤2: [操作描述] → [状态变化]
...
步骤N: [操作描述] → [状态变化]
```

### 2. 测试用例设计原则

#### 2.1 功能模块与测试点划分（必须遵守）

**功能模块划分**：

- 按业务域划分，每个模块有独立的业务边界
- 命名使用业务域名称，如"设备绑定管理"、"用户权限管理"
- 模块数量应该依据需求文档客观分析，得到实际业务功能模块的数量，避免根据个人经验或假设来划分模块

**测试点划分**：

- 按需求中的实际子功能/操作划分，有多少写多少
- 测试点是模块下的具体操作，可能1个也可能10个
- 禁止测试点名称与功能模块名称相同

**正确示例**：

```
设备绑定管理/     # 功能模块
├── 在线绑定.md   # 测试点1
├── 离线预绑定.md # 测试点2
└── 设备解绑.md   # 测试点3
```

**错误示例**：

```
设备绑定管理/     # 功能模块
└── 设备绑定.md   # ❌ 与模块名重复，划分不清晰
```

**检验**：生成后检查功能模块名是否与任何测试点名称重复

#### 2.2 核心原则

1. **必须同时覆盖正向和反向、异常场景**
   - 正向场景：需求中明确描述的正常业务流转（Happy Path）。
   - 反向场景：需求中明确提及的反向业务流转（如驳回、重试、返回等）。
   - 异常场景：需求中明确提及的异常分支（如驳回、报错、状态拦截），以及**基于需求约束隐含的边界和异常流**。
   - **注意**：只测试需求中明确提到的功能约束，不添加需求未提及的功能测试；不进行详细的输入校验测试（除非需求明确要求）。
2. **操作步骤与预期结果严格对应**
   - 测试步骤的序号必须与预期结果的序号保持**严格的一致性和对应关系**。
   - 如果某个步骤没有需要检查的预期结果，在预期结果中该序号应留空不显示。
   - **示例**：如果操作步骤有 1 到 6 步，但只有第 3 步和第 6 步有明确的检查点，则预期结果的结构应为：
     ```
     [预期结果]
     3. [对应第3步的预期结果]
     6. [对应第6步的预期结果]
     ```
3. **按照流程步骤组织测试用例**
   - 测试用例按照业务流程步骤的执行顺序排列
   - 每个流程步骤对应一个或多个测试用例
   - 测试用例的执行顺序与业务流程一致
4. **测试用例数量原则**
   - 功能模块、测试点、用例数都应按需求实际情况**客观分析**
   - 依据设计方法分析需要覆盖哪些场景，就生成多少用例
   - 不按需求复杂性限制用例数量
5. **优先级判定规则（关键）**
   - **P0（10-15%）**：仅核心功能正向流程。判定方法：该功能失效是否导致核心业务中断？是→P0
   - **P1（20-30%）**：基本功能 + 常见异常场景。判定方法：功能可用但非核心流程？是→P1
   - **P2（35-45%）**：边界值 + 异常流 + 网络异常 + 权限限制。判定方法：是否验证约束条件的边界或违反情况？是→P2
   - **P3（10-15%）**：UI展示类（列表显示、状态样式、按钮可见性、页面布局）+ 极端场景 + 体验优化。判定方法：是否仅为UI展示、极端边界或体验优化？是→P3
   - **强制约束**：P0+P1≤40%，P2应占最大比例
   - **常见错误**：将UI展示、状态样式、列表显示等标记为P0或P1（应为P3）；将边界值校验、异常流标记为P1（应为P2）
     **迭代划分流程**（推荐）：
   1. **初步划分**：功能→P1，错误/边界→P2，非功能→P3，UI展示→P3
   2. **提升降级**：根据"重要性"调整（重要=用户使用频率高+缺陷历史多+业务影响大）
      - "不重要"的功能测试 → 降级为P2
      - "重要"的错误/边界测试 → 升级为P1
      - "重要"的非功能测试 → 升级为P2
   3. **挑出P0**：从高优先级中挑出"严重"的（fail会阻碍其他用例验证）升级为P0
   4. **动态调整**：随着项目进展，根据缺陷分布和用户反馈持续优化优先级
6. **设计方法选择（基于约束条件灵活选择）**
   - 有输入参数/取值范围 → 等价类划分 + 边界值分析
   - 有多条件组合判断 → 判定表法 + 因果图法
   - 有状态流转 → 状态迁移法
   - 有多参数组合 → 正交实验法
   - 有明确流程步骤 → 场景法
   - **至少选择1种，核心功能建议2-3种，复杂功能建议3-5种**
   - 根据功能约束条件选择最合适的方法，而非强制要求固定数量
7. **场景覆盖要求**
   - 每个核心功能必须覆盖：正向流程 + 边界值 + 至少1个异常场景
   - 需求中明确提及的状态/条件类约束，必须覆盖允许状态和禁止状态
   - 需求中明确提及的权限类功能，必须覆盖有权限操作和无权限拦截

### 3. 编写测试用例

每个用例必须包含：

- 具体的测试步骤（包含实际数据）
- 可验证的预期结果
- 对应的业务流程步骤

#### 3.1 用例结构

```markdown
## [P0] 用例标题
[测试类型] 功能
[前置条件] 前置条件描述
[测试步骤] 1. 步骤1。2. 步骤2。3. 步骤3
[预期结果] 3. 预期3
```

#### 3.2 字段规范

| 字段   | 规范                                         | 示例                                                      |
| ---- | ------------------------------------------ | ------------------------------------------------------- |
| 标题   | \[反向(如果有)]用自然语言描述清楚测试用例的目的                 | `## [P0] 使用正确账号名密码成功登录系统`                               |
| 测试类型 | 12种枚举之一，默认"功能"                             | `[测试类型] 功能`                                             |
| 前置条件 | 分号分隔，只写必要条件                                | `[前置条件] 管理员已登录`                                         |
| 测试步骤 | `1. xxx。2. xxx` 格式，包含具体数据                  | `[测试步骤] 1. 打开工单管理页面。2. 点击创建工单按钮。3. 输入信息并保存`             |
| 预期结果 | `1. xxx。2. xxx` 格式，与测试步骤序号严格一一对应，无需检查的步骤留空 | `[预期结果] 1.工单列表页面数据正常显示。2.正确打开创建工单页面。3. 工单创建成功，工单状态为待分配` |

### 4. 生成质量评审报告

生成测试用例后，**必须先执行引导错误过滤检查**，只有通过后才能生成正式的质量评审报告并导出用例。

#### 4.1 引导错误过滤检查（必须先执行）

以下任何一项存在则判定为**不合格**，需要返回**重新生成**：

| 引导错误      | 检查方法                   | 判定结果 |
| --------- | ---------------------- | ---- |
| 数据占位符     | 搜索 `{`、`}`、包含`xxx`的占位符 | 不合格  |
| 预期模糊      | 搜索"功能正常"、"显示正确"、"正常工作" | 不合格  |
| 步骤不对应     | 逐条对比测试步骤序号与预期结果序号      | 不合格  |
| P0+P1>50% | 统计P0+P1占比              | 不合格  |

**检查流程**：

```
1. 遍历所有用例，检查是否有占位符 → 如有，标记具体用例编号
2. 遍历所有用例，检查预期是否模糊 → 如有，标记具体用例编号
3. 遍历所有用例，核对步骤/预期序号对应 → 如有不符，标记
4. 统计优先级分布 → 如>50%，标记
5. 输出检查结论：合格/不合格 + 具体问题清单
```

> **一票否决制**：只要存在任何引导错误，质量评审结论即为"不合格"，用例**全盘不用**，需要返回重新生成。

**重新生成逻辑**：

```
生成用例 → 质量评审 → 判定结果
                                ↓
                    ┌────────────┴────────────┐
                    ↓                         ↓
              评审合格                    评审不合格
                    ↓                         ↓
            生成质量评审报告            记录问题，返回重新生成
                    ↓                         ↓
              执行导出脚本            重新生成用例 → 重新评审（循环）
```

- 评审合格：生成完整质量评审报告 → 执行导出脚本
- 评审不合格：**用例全盘不用**，标记具体问题 → 返回阶段6重新生成 → 重新执行质量评审 → 直到评审通过

#### 4.2 统计概览

- 测试用例总数
- 业务流程步骤数
- 优先级分布（P0/P1/P2/P3 数量及占比）
- 测试类型分布

#### 4.3 引导错误检查结果

- 检查结论：合格/不合格
- 发现的问题清单（如有）

#### 4.4 六大维度评估

- **PRD 覆盖度**：是否覆盖所有功能点和场景（正向+异常）
- **用例冗余性**：是否存在逻辑完全相同、重复或价值低的用例
- **步骤清晰度**：步骤是否具体可执行（拒绝"测试功能"等宽泛描述）
- **预期明确性**：预期结果是否具备明确的可验证物（拒绝"功能正常"等模糊描述）
- **场景完整性**：是否包含边界值、异常流等完整场景
- **优先级合理性**：P0/P1/P2/P3 划分是否合理（核心流程必须为P0）

#### 4.5 覆盖率量化

- **需求覆盖率**：功能需求≥95%，非功能需求≥80%
- **测试类型覆盖率**：至少覆盖3种测试类型
- **设计方法覆盖率**：至少应用1种用例设计方法，选择与功能约束匹配
- **边界值覆盖率**：100%覆盖min-1/min/max+1
- **异常场景覆盖率**：至少覆盖80%可能的异常情况

#### 4.6 测试用例质量评估

- **需求覆盖度**：是否覆盖所有功能点和场景（正向+异常+边界）
- **设计方法应用**：是否正确应用等价类、边界值、因果图等方法
- **场景完整性**：正向、反向、边界、异常场景是否齐全
- **可执行性**：步骤是否清晰、预期是否明确可验证
- **优先级合理性**：P0/P1/P2/P3分布是否符合标准（P0:10-15%, P1:20-30%, P2:35-45%, P3:10-15%）

#### 4.7 重复检测

- 重复用例（相似度≥90%）
- 语义相似用例（相似度70%-89%）

#### 4.8 待确认需求清单

- 需求未明确但测试方法要求覆盖的场景
- 格式：| 用例编号 | 测试场景 | 待确认点 | 建议处理方式 | 风险等级 |

#### 4.9 可扩展用例建议

- 根据需求文档和当前测试用例，建议可以扩展的测试用例

#### 4.10 结论

- 总体评分（百分制）
- 改进建议
