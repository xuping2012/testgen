# 测试用例格式规范

## Markdown 文本协议

### 基本结构

每个测试点生成一个独立的 `.md` 文件，包含多个测试用例。

```markdown
## [P1] 用例标题
[测试类型] 功能
[前置条件] 前置条件描述
[测试步骤] 1. 步骤1。2. 步骤2
[预期结果] 1. 预期1。2. 预期2

## [P2] 用例标题
[测试类型] 功能
[前置条件] 前置条件描述
[测试步骤] 1. 步骤1。2. 步骤2
[预期结果] 1. 预期1。2. 预期2
```

### 字段详解

#### 1. 优先级（Priority）

**格式**：`[P0]` 或 `[P1]` 或 `[P2]` 或 `[P3]`

**定义**：

| 优先级 | 适用场景      | 生成策略          |
| --- | --------- | ------------- |
| P0  | 核心功能正向流程  | 最常见的成功场景，必测   |
| P1  | 基本功能正向流程  | 其他有效等价类，常规测试  |
| P2  | 核心异常场景    | 关键失败场景，必须覆盖   |
| P3  | 边界条件、低频场景 | 边界值测试，不常见但需覆盖 |

**优先级判定树**：

```
是否核心功能？
├─ 是 → 正向用例 → P0
│      反向用例 → P1（常见异常）/ P2（边界/极端异常）
├─ 否（基本功能）→ 正向 → P1
│                反向 → P2（常见异常）/ P3（低频异常）
└─ 否（不常用）→ P3
```

#### 2. 测试类型

**格式**：`[测试类型] 类型名称`

**枚举值（12种）**：

- 功能（含UI交互功能属于功能）
- 兼容性（含UI不同设备、浏览器、操作系统属于兼容性）
- 易用性（含UI交互易用性属于易用性）
- 性能
- 稳定性
- 安全性
- 可靠性
- 效果（AI类、资源类）
- 效果（硬件器件类）
- 可维护性
- 可移植性
- 埋点

**注意事项**：

- 不能写"功能测试"，必须是"功能"
- 括号必须是中文括号
- 大小写敏感
- 默认为"功能"

#### 3. 前置条件

**格式**：`[前置条件] 条件描述`

**要求**：

- 分号分隔多个条件
- 只写必要条件
- 包含环境准备和数据准备

**示例**：

```
[前置条件] 用户已注册；账号未锁定；系统已登录
```

#### 4. 测试步骤

**格式**：`[测试步骤] 1. 步骤1。2. 步骤2。3. 步骤3`

**要求**：

- 使用中文句号分隔
- 编号从1开始，连续递增
- 步骤必须具体可执行
- 包含具体的测试数据

**示例**：

```
[测试步骤] 1. 打开登录页面，输入用户名test_user。2. 输入密码Test1234。3. 点击登录按钮
```

#### 5. 预期结果

**格式**：`[预期结果] 1. 结果1。2. 结果2。3. 结果3`

**要求**：

- 使用中文句号分隔
- 编号从1开始，连续递增
- 必须与测试步骤数量一致
- 结果必须明确可验证

**示例**：

```
[预期结果] 1. 登录请求成功。2. 跳转到/home页面。3. 页面顶部显示"欢迎，test_user"
```

### 格式规则

#### 1. 步骤与结果对应

**错误示例**：

```
[测试步骤] 1. 打开登录页。2. 输入用户名。3. 输入密码。4. 点击登录。
[预期结果] 1. 登录成功。2. 跳转首页。
```

❌ 步骤4个，结果2个，不匹配

**正确示例**：

```
[测试步骤] 1. 打开登录页，输入用户名test_user。2. 输入密码Test1234。3. 点击登录。
[预期结果] 1. 页面正常显示。3. 登录成功，跳转到首页。
```

✅ 步骤3个，结果2个，第二步只有录入没有预期结果，第三步登录才有交互.

#### 2. 测试数据要求

**错误示例**：

```
[测试步骤] 1. 输入用户名{username}。2. 输入密码{password}。
```

❌ 使用占位符

**正确示例**：

```
[测试步骤] 1. 输入用户名test_user。2. 输入密码Test1234。
```

✅ 使用具体数据

#### 3. 预期结果可验证

**错误示例**：

```
[预期结果] 1. 系统正常处理。2. 提示错误信息。
```

❌ 结果不明确

**正确示例**：

```
[预期结果] 1. 提示"密码长度不足，请输入8-20位密码"。2. 停留在登录页
```

✅ 结果明确可验证

#### 4. 用例标题规范

**核心原则**：简洁明了、通俗易懂、具体明确

##### 4.1 长度要求

| 长度范围       | 评价    | 建议           |
| ---------- | ----- | ------------ |
| **15-30字** | ✅ 最佳  | 推荐使用，能清晰表达意图 |
| **10-15字** | ⚠️ 偏短 | 可接受，但建议补充描述  |
| **<10字**   | ❌ 太短  | 不推荐，缺乏描述性    |
| **>40字**   | ❌ 过长  | 不推荐，信息冗余     |

##### 4.2 标题结构公式

**正向场景**：

```
[前置条件/角色] + 操作动作 + 成功结果
```

示例：

- `管理员成功绑定在线设备到房间`
- `租客在合同有效期内操作空调设备成功`
- `使用正确账号密码登录系统成功`

**反向场景**：

```
[反向] + 错误条件 + 失败结果
```

示例：

- `[反向] 使用错误密码登录系统失败`
- `[反向] SN码格式错误导致设备绑定失败`
- `[反向] 租客越权访问其他房间设备被拦截`

**边界场景**：

```
操作对象 + 边界值描述 + 验证点
```

示例：

- `空调温度设置为16度(温度最小值)`
- `SN码长度为22位(低于最小长度23位)`

##### 4.3 标题编写禁忌

❌ **禁止出现**：

- 以"控制"、"操作"、"测试"、"验证"结尾（冗余、不具体）
- 包含"功能正常"、"正常工作"等模糊描述
- 使用占位符如`{username}`、`{id}`
- 纯数字或无意义字符

❌ **不合格标题示例**：

| 不合格标题    | 字数 | 问题         | 改进建议            |
| -------- | -- | ---------- | --------------- |
| `空调开关控制` | 6字 | 太短+以"控制"结尾 | `管理员成功开启空调`     |
| `SN码为空`  | 5字 | 太短+缺乏场景    | `输入空SN码绑定设备失败`  |
| `取消解绑操作` | 7字 | 以"操作"结尾    | `取消解绑后设备保持绑定状态` |
| `功能正常`   | 4字 | 完全无意义      | `管理员成功绑定设备到房间`  |

✅ **合格标题示例**：

| 合格标题                | 字数  | 优点     |
| ------------------- | --- | ------ |
| `正常绑定在线设备到房间`       | 11字 | 简洁明确   |
| `管理员开启调试模式`         | 8字  | 清晰易懂   |
| `租客在合同有效期内操作空调设备成功` | 17字 | 完整描述   |
| `空调温度设置为最小值16度`     | 12字 | 包含边界值  |
| `[反向] 输入错误密码登录失败`   | 14字 | 明确反向场景 |
| `[反向] SN码格式错误绑定失败`  | 14字 | 明确异常原因 |
| `设备离线时无法操作开关`       | 11字 | 描述异常场景 |
| `订单支付超时自动取消`        | 10字 | 描述时间边界 |

##### 4.4 标题质量检查清单

生成用例时，标题必须通过以下检查：

- [ ] **长度检查**：10-40字（最佳15-30字）
- [ ] **结构检查**：符合正向/反向/边界场景公式
- [ ] **语义检查**：通俗易懂，非测试人员也能理解
- [ ] **结尾检查**：不以"控制"、"操作"、"测试"结尾
- [ ] **唯一性检查**：与同模块其他用例标题不重复
- [ ] **优先级匹配**：P0/P1为正向场景，P2为边界/异常，P3为UI展示

#### 5、错误用例 - 重复用例

**错误示例**：

```markdown
## [P2] OTA升级-检测有最新固件版本
[测试类型] 功能
[前置条件] 已进入网关详情页，点击OTA升级按钮
[测试步骤] 1. 点击OTA升级按钮。2. 查看页面弹窗。3. 查看弹窗内容
[预期结果] 1. 弹出OTA升级弹窗。2. 页面检测到有最新固件版本, 显示升级按钮。3. 显示当前版本号和最新版本号。

## [P2] OTA升级-显示升级按钮
[测试类型] 功能
[前置条件] OTA升级弹窗检测到有最新固件版本
[测试步骤] 1. 查看OTA升级弹窗。2. 查看升级按钮显示
[预期结果] 1. 显示"立即升级"按钮。2. 升级按钮样式清晰可见,升级按钮可点击
```

**问题分析**：

1. **第二个用例是第一个用例的子集**
   - 第一个用例已经验证了"显示升级按钮"
   - 第二个用例重复验证相同内容
2. **预期结果已覆盖**
   - 第一个用例的预期结果3：显示升级按钮
   - 第二个用例完全重复

**正确示例**：

```markdown
## [P2] OTA升级-检测有最新固件版本
[测试类型] 功能
[前置条件] 已进入网关详情页，当前固件版本v1.0
[测试步骤] 1. 点击OTA升级按钮。
[预期结果] 1. 弹出OTA升级弹窗，显示当前版本v1.0和最新版本v2.0，显示"立即升级"按钮。

## [P2] OTA升级-无新版本时不显示升级按钮
[测试类型] 功能
[前置条件] 已进入网关详情页，当前固件版本已是最新
[测试步骤] 1. 点击OTA升级按钮。
[预期结果] 1. 弹出OTA升级弹窗，显示当前版本已是最新版本，不显示升级按钮，显示"确定"按钮。
```

#### 6、错误用例 - 预期结果不可验证

**错误示例**

```markdown
## [P1] 验证支付跳转功能
[测试类型] 功能
[前置条件] 用户已登录，购物车有商品
[测试步骤] 1. 进入收银台，选择支付方式，点击支付。
[预期结果] 1. 系统正常处理支付请求。
```

**问题分析**：

1. **预期结果模糊**
   - "系统正常处理支付请求"无法验证
   - 没有明确的成功标准

**正确示例**：

```markdown
## [P1] 选择支付宝支付成功
[测试类型] 功能
[前置条件] 用户已登录，购物车有商品，订单金额99.99元
[测试步骤] 1. 进入收银台，选择支付宝，点击支付，在支付宝页面完成支付
[预期结果] 1. 跳转到支付成功页面，显示订单已支付，订单状态更新为"已支付"
```

### Xmind 导出格式

#### 层级结构

```
测试用例 (根节点)
├── 测试项 (ITEM)
│   ├── 测试点 (POINT)
│   │   ├── 测试用例 1
│   │   │   └── 前置条件
│   │   │        └── 测试步骤
│   │   │             └── 预期结果
│   │   │                  └── 优先级
│   │   └── 测试用例 2
│   │       ├── ...
│   └── ...
└── ...
```

#### 节点类型

| 节点类型 | class 属性     | 说明     |
| ---- | ------------ | ------ |
| 根节点  | topic        | 测试用例   |
| 测试项  | testitem     | 功能模块   |
| 测试点  | testpoint    | 具体测试点  |
| 测试用例 | testcase     | 单个用例   |
| 前置条件 | precondition | 用例前置条件 |
| 测试步骤 | steps        | 操作步骤   |
| 预期结果 | expected     | 期望结果   |
| 优先级  | priority     | 用例优先级  |

### 目录结构

```
test-case/
├── plan.md                    # 测试计划（计划驱动模式）
├── {功能模块}/                 # 功能模块目录（一级）
│   ├── {测试点1}.md            # 测试点用例（二级，至少2个）
│   ├── {测试点2}.md            # 测试点用例
│   └── ...
├── all_cases.md               # 所有用例汇总
├── {需求}_评审报告_{时间戳}.md        # 测试用例评审报告，md格式
├── {需求}_测试用例_{LLM_NAME}_{时间戳}.xlsx     # Excel格式
└── {需求}_测试用例_{LLM_NAME}_{时间戳}.xmmind    # Xmind格式
```

### 功能模块与测试点划分规范

**命名原则**：

| 层级   | 命名规则            | 示例            |
| ---- | --------------- | ------------- |
| 功能模块 | 业务域名称           | 设备绑定管理、用户权限管理 |
| 测试点  | 具体操作名称，禁止与模块名重复 | 在线绑定、设备解绑     |

**正确结构**：

```
test-case/
├── 设备绑定管理/              # 模块：设备生命周期
│   ├── 在线绑定.md           # 测试点（按需求实际子功能）
│   ├── 离线预绑定.md         # 测试点
│   └── 设备解绑.md           # 测试点
├── PMS智能家居管理/           # 模块：PMS端设备管理
│   └── 智能家居列表.md       # 测试点（只有1个也正常）
```

**错误结构（禁止出现）**：

```
test-case/
├── 设备绑定管理/
│   └── 设备绑定.md           # ❌ 模块名=测试点，重叠
├── 空调控制功能/
│   └── 空调控制.md           # ❌ 模块名=测试点，重叠
```

**核心原则**：测试点数量实事求是，按需求中的实际子功能决定，关键是名称不能与模块重复。

**合并规则**

**all\_cases.md 生成规则**：

1. 按 ITEM 分组
2. 每个 ITEM 下按 POINT 排序
3. 添加分隔标记：
   - `# ITEM 名称`（功能模块）
   - `## POINT 名称`（测试点）

**示例**：

```markdown
# 用户登录功能

## 用户名密码登录

## [P1] 输入有效账号密码登录成功
[测试类型] 功能
[前置条件] 用户已注册
[测试步骤] 1. 输入用户名test。2. 输入密码Test123。3. 点击登录。
[预期结果] 3. 登录成功，跳转首页。

## 第三方登录

## [P1] 使用微信登录成功
...
```

### 命名规范

#### 文件命名

**ITEM 目录**：

- 使用中文功能名称
- 简洁明确
- 示例：`用户登录功能`、`订单管理`、`支付模块`

**POINT 文件**：

- 使用中文测试点名称
- 简洁明确
- 示例：`用户名密码登录.md`、`第三方登录.md`

#### 导出文件命名

- Excel：`all_cases_测试用例_{LLM_NAME}_{时间戳}.xlsx`
- Xmind：`all_cases_思维导图_{LLM_NAME}_{时间戳}.xmmind`
- 汇总：`all_cases.md`
- 报告：`{需求}_测试用例_评审报告_{时间戳}.md`

### 特殊字符处理

#### 需转义的字符

在Excel导出时，以下字符需要特殊处理：

- 单引号 `'` → 双单引号 `''`
- 双引号 `"` → 保持或转义
- 换行符 `\n` → Excel自动换行
- 制表符 `\t` → Excel制表

#### 示例

```
原始：提示"密码不能为空"
Excel：提示"密码不能为空"
```

### 验证规则

#### 必填字段

- [ ] 优先级（P0-P3）
- [ ] 测试类型（12种之一）
- [ ] 测试步骤（至少1步）
- [ ] 预期结果（至少1个）

#### 格式检查

- [ ] 优先级格式正确：`[P0]`
- [ ] 测试类型在枚举中
- [ ] 步骤编号从1开始，连续递增
- [ ] 结果编号从1开始，连续递增
- [ ] 步骤数量 不一定等于 结果数量，预期结果应对应有结果的操作步骤序号
- [ ] 不使用占位符
- [ ] 预期结果可验证

#### 内容检查

- [ ] 测试数据具体
- [ ] 步骤可执行
- [ ] 结果可验证
- [ ] 无重复用例
- [ ] 标题清晰明确
