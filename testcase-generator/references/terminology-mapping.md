## 目的
统一 `testcase-generator` 技能与 `theory.md` 的术语体系，消除理解偏差，确保技能自检与理论指导一致。

## 1. 技能六大维度 ↔ 理论章节映射

| 技能维度 | 理论支撑章节 | 核心要点 | 检查清单 |
|---------|-------------|---------|---------|
| **PRD覆盖度** | 第8章 覆盖率量化标准 | 功能覆盖100%、规则覆盖100%、场景覆盖≥95% | - 是否覆盖所有P0/P1功能点？<br>- 是否覆盖所有业务规则？<br>- 场景覆盖度是否≥95%？ |
| **用例冗余性** | 第1章 等价类划分法 | 一次只测一个等价类，避免重复 | - 是否存在逻辑完全相同的用例？<br>- 每个用例是否有独立验证价值？ |
| **步骤清晰度** | 第12章 用例质量评估 | 3-10步，具体到界面元素+输入数据 | - 步骤是否包含具体输入数据？<br>- 步骤数是否在3-10之间？<br>- 是否避免模糊动词（查看、测试）？ |
| **预期明确性** | 第12章 用例质量评估 | 可验证，拒绝模糊描述 | - 预期结果是否可量化判断？<br>- 是否避免"功能正常"等模糊词？<br>- 是否标注`[需确认需求]`？ |
| **场景完整性** | 第9、10、11章 | 四类场景分布 + 边界矩阵 + 异常框架 | - 正向场景占比25-30%？<br>- 反向场景占比30-35%？<br>- 边界值场景占比20-25%？<br>- 异常场景占比10-20%？ |
| **优先级合理性** | 第13章 用例规划 | P0占比10-15%（所有功能复杂度统一） | - P0是否为核心主流程？<br>- P0占比是否在10-15%范围内？<br>- 优先级分布是否合理？ |

## 2. 技能12种测试类型 ↔ 理论方法映射

> 注：技能定义的12种测试类型是"测试类型"分类，而场景分类（正向/反向/边界/异常）是另一维度，两者正交。

| 技能测试类型 | 理论方法 | 应用场景 | 设计原则 |
|-------------|---------|---------|---------|
| **功能** | 等价类划分法 + 场景法 | 主流程验证 | 覆盖所有正向等价类 |
| **兼容性** | 边界值分析法 + 异常场景框架 | 多浏览器/设备/分辨率适配 | 主流环境全覆盖，扩展环境抽样 |
| **易用性** | 场景法 + 错误推测法 | UI交互体验、提示信息友好性 | 从用户视角设计操作路径 |
| **性能** | - | 响应时间、吞吐量、并发能力 | 基于需求指标设计负载场景 |
| **稳定性** | 异常场景设计框架 | 长时间运行、高负载容错 | 持续运行+资源监控 |
| **安全性** | 错误推测法 + 因果图法 | 认证、授权、数据保护、漏洞防护 | 渗透测试思路+权限矩阵 |
| **可靠性** | 状态迁移法 + 因果图法 | 数据一致性、幂等性、重试机制 | 异常恢复+数据完整性验证 |
| **可维护性** | - | 日志完整性、监控指标、配置管理 | 运维视角验证可观测性 |
| **可移植性** | - | 多环境部署、配置迁移 | 跨环境安装部署验证 |
| **效果（AI类、资源类）** | - | 算法准确率、资源消耗 | 基于算法指标设计验证集 |
| **效果（硬件器件类）** | 正交试验法 | 硬件性能指标、稳定性、兼容性 | 多型号/批次抽样 |
| **埋点** | 判定表法 | 数据采集完整性、准确性、时效性 | 事件触发条件全覆盖 |

> 标注"-"的理论方法表示 theory.md 当前未覆盖该测试类型的详细方法论，需结合行业实践补充。

## 3. 关键术语对照表

| 技能术语 | theory.md术语 | 说明 |
|---------|--------------|------|
| PRD覆盖度 | 覆盖率量化标准 | 两者含义一致，统一使用"PRD覆盖度" |
| 场景完整性 | 场景类型分布策略 | 技能强调"完整"，理论强调"分布"，内涵一致 |

## 4. 需求明确性判断标准

### 明确的需求（必须覆盖）
- ✅ 需求文档明确描述的功能流程
- ✅ 需求文档明确给出的约束条件（如长度范围、必填项）
- ✅ 需求文档明确描述的错误码和异常处理

### 待确认的需求（应测试并标注）
- ⚠️ 需求给出约束但未明确异常处理（如长度8-16位，但未说7位如何处理）
- ⚠️ 需求隐含但未明确表述的规则（如必填项为空）
- ⚠️ 测试设计方法要求的场景（如边界值、反向场景）

### 不测试的场景
- ❌ 需求完全未提及的功能
- ❌ 与需求无关的技术异常（如需求未要求并发，则不测试并发）

## 6. 使用指南

### 技能自检时
1. 遇到理解偏差，查阅本映射表的"技能维度 ↔ 理论章节映射"
2. 不确定是否该测试某场景，查阅"需求明确性判断标准"
3. 评审报告某章节不知如何填写，查阅"质量评审报告结构映射"

### 理论应用时
1. 设计用例时，参考"技能12种测试类型 ↔ 理论方法映射"选择合适方法
2. 评估用例质量时，参考"关键术语对照表"统一表述
3. 处理需求不明确场景时，遵循"需求驱动测试原则"（第14章）
