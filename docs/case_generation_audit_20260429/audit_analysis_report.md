## 一、生成过程日志问题

### 1.1 FTS5搜索语法错误（严重程度：⚠️ 中）

**日志位置**: 行65、67

**现象**: RAG召回时，需求原文含中文标号"1、2、3、"导致SQLite FTS5报错：

```
near "'1、账号名只能是邮箱地址 2、密码长度8-18 强校验 3、邮箱验证码校验'": syntax error
```

**根因**: 将需求原文直接作为FTS5查询字符串，未做预处理——未剥离中文序号标点、未转义FTS5敏感字符（引号、括号等）。

**影响**: 两次FTS5执行失败后系统自动expand检索范围(k=5→10)作为兜底，但召回质量可能下降，因为向量搜索被迫承担全部检索职责。

### 1.2 Prompt模板残留示例占位符（严重程度：⚠️ 中）

**日志位置**: 行82-83

**现象**: `case_generation`模板渲染时报缺少变量 `username`, `xxx`, `password`：

```
[Prompt渲染] 模板 case_generation 缺少变量: ['username', 'xxx', 'password']
```

**根因**: 模板中残留了示例/调试用的占位符变量，未被清理或替换为实际业务变量。

**影响**: 不影响生成结果（缺失变量被忽略），但属于模板维护缺陷，可能导致：

- 渲染后的Prompt中出现空位或乱码
- 模板意图与实际数据不一致

### 1.3 LLM响应JSON解析反复失败（严重程度：🔴 高）

**日志位置**: 行102、120、127、140、147、154、159

**现象**: 所有5次主生成+6次补充生成的LLM调用，**直接JSON解析全部失败**（`Expecting value: line 1 column 1`），必须靠"从Markdown代码块提取JSON"兜底解析。

**根因**: LLM稳定返回 ` ```json ... ``` ` 格式包裹的响应，但系统解析逻辑优先尝试裸JSON解析，每次先走失败路径再fallback到代码块提取。

**影响**:

- 每次解析多走一个无意义的失败路径，增加处理延迟
- 错误日志堆积，干扰真正异常的排查
- 如果LLM偶尔返回裸JSON，两种解析路径的行为不一致可能产生差异

### 1.4 首轮覆盖率仅33%（严重程度：⚠️ 中）

**日志位置**: 行114

**现象**: 首轮生成12条用例只覆盖3/9测试点(33.33%)，需要6轮补充生成才达到100%覆盖。

**根因**: 主生成Prompt的覆盖引导不够强，LLM倾向于集中生成部分测试点（如"正常流程"、"边界值"、"异常处理"），而对评审补充的测试点（如"验证码时效性"、"国际化邮箱"、"频率限制"）覆盖不足。

**影响**: 补充生成机制虽然最终补齐了覆盖率，但每轮补充都是独立调用LLM，导致同一测试点产生两套用例（主生成+补充生成），引发大量重复。

## 四、流程优化建议

### 4.1 需求分析阶段优化

#### 问题：分析结果遗漏或偏差导致下游全链路缺陷

**建议A：建立需求-测试点映射矩阵**

在需求分析完成后，系统应自动生成一份**需求条款→测试点**的映射矩阵，确保每条需求条款至少有1个测试点对应：

```
需求条款1: "账号名只能是邮箱地址"
  → 测试点: 非邮箱格式校验 / 已注册邮箱校验 / 邮箱格式变体

需求条款2: "密码长度8-18，强校验"
  → 测试点: 长度边界(7/8/18/19) / 强校验复杂度(纯数字/纯字母/无特殊字符/无大写)

需求条款3: "邮箱验证码校验"
  → 测试点: 正确验证码 / 错误验证码 / 过期验证码 / 验证码安全机制
```

**实现方式**：修改`requirement_analysis` Prompt模板，要求LLM输出结构化的需求条款-测试点映射，而非松散的测试点列表。

**建议B：引入需求条款完整性自检**

在分析结果返回前，系统自动执行反向校验：

1. 从分析结果中提取所有测试点
2. 将测试点映射回需求条款
3. 检查是否存在"无测试点对应的需求条款"
4. 如存在，自动补充缺失测试点或向用户发出警告

### 4.2 模块评审阶段优化

#### 问题：评审建议未被下游采纳，评审与生成脱节

**建议C：评审结论必须结构化传递至生成阶段**

当前评审结果只生成文字描述的评审建议（如"验证码频率限制测试应提升至P1"），但这些建议在Phase 2生成时**未被结构化消费**。

改为：

- 评审结果输出结构化JSON，包含 `priority_adjustments`（优先级调整建议）、`missing_test_points`（缺失测试点）、`assumption_warnings`（假设风险提示）
- Phase 2生成时，将评审结论的 `priority_adjustments` 直接注入生成Prompt，确保LLM采纳优先级调整
- 将 `missing_test_points` 直接作为强制覆盖项，而非等覆盖率检查后再补充

**建议D：评审阶段标注"需求来源条款"**

每个测试点应标注其对应的需求条款编号。当评审发现某测试点偏离需求原文时（如TC\_000012的SQL注入），应标记为"偏离需求"，并在生成阶段予以过滤或降级。

### 4.3 用例生成阶段优化

#### 问题：主生成+补充生成产生大量语义重复

**建议E：统一生成而非分批补充**

当前流程：

1. 主生成 → 覆盖率33%
2. 发现未覆盖测试点 → 逐个补充生成 → 每次独立调用LLM
3. 两套用例合并 → 语义重复7对

改为**一次性生成**策略：

- 将所有测试点（包括评审补充的）统一放入一个生成Prompt
- 在Prompt中明确要求"每个测试点生成1\~2条用例，不得为同一测试点生成多套用例"
- 如果测试点数量过多导致Prompt过长，按模块分批生成，但**每批内包含完整测试点列表**，避免跨批重复

**建议F：补充生成时传入已生成用例清单**

如果必须保留补充生成机制，则在补充生成Prompt中**明确列出已生成的用例标题和测试点**，要求LLM：

- 仅为**未覆盖的测试点**生成用例
- 不得与已有用例重复
- 明确标注补充用例与已有用例的区别

### 4.4 质检去重阶段优化

#### 问题：文本级去重无法检出语义重复

**建议G：升级为语义级去重**

当前去重算法仅做文本精确匹配（比较用例标题/步骤的字符串相似度），无法识别以下语义重复：

- "用户输入18位密码边界值验证" vs "验证18位密码注册成功" → 文本不同但语义相同
- "连续5次输入错误验证码后账户被锁定" vs "连续5次验证码错误触发账号锁定" → 文本微调但测试意图完全相同

升级方案（分阶段实施）：

**第一阶段（低成本）**：关键词+测试点交叉去重

1. 提取每条用例的测试点标签
2. 如果两条用例的测试点标签相同，且核心操作步骤的语义意图相同（如都测"18位密码"），标记为疑似重复
3. 对疑似重复对，取质量评分更高的那条保留

**第二阶段（中成本）**：LLM辅助语义去重

1. 对疑似重复对，调用LLM判定是否语义相同
2. LLM返回去重决策：保留哪条、合并、或两条都保留（如有细微差异）
3. 此调用应使用低成本模型（temperature=0）以节省开销

**第三阶段（高成本）**：向量相似度去重

1. 将每条用例的"测试标题+操作步骤+预期结果"编码为向量
2. 计算用例间向量相似度
3. 相似度超过阈值的对标记为疑似重复

### 4.5 LLM响应解析优化

#### 问题：每次解析先失败再fallback

**建议H：调整解析优先级**

当前解析顺序：

1. 尝试直接JSON解析 → 失败
2. 尝试从Markdown代码块提取JSON → 成功

改为：

1. 先检测响应是否包含Markdown代码块标记（`json 或 ` ）
2. 如包含，直接提取代码块内容再解析JSON
3. 如不包含，再尝试直接JSON解析
4. 两种路径都不成功时，才记录错误日志

**实现方式**：修改 `generation_service.py` 中的LLM响应解析方法，将代码块检测前置。

### 4.6 RAG召回优化

#### 问题：FTS5查询语法错误

**建议I：FTS5查询预处理**

在将需求文本传入FTS5搜索前，执行预处理：

1. 剥离中文序号标点（"1、"、"2、"等）
2. 转义FTS5敏感字符（引号、括号等）
3. 将长文本拆分为多个短语，分别查询后合并结果
4. 或者仅使用向量搜索路径，完全绕过FTS5（当查询文本复杂时）

### 4.7 Prompt模板维护优化

#### 问题：模板残留示例占位符

**建议J：模板变量校验机制**

在Prompt模板保存/更新时：

1. 自动扫描模板中的变量占位符（如 `{variable_name}`）
2. 与已知业务变量清单交叉比对
3. 未在业务变量清单中出现的占位符标记为"疑似残留"，提示用户确认
4. 渲染时，对缺失变量不再只打印warning，而是用空字符串替换或用默认值填充，避免模板中出现裸露的占位符标记

### 4.8 优先级自动化调整

#### 问题：评审建议的优先级调整未被采纳

**建议K：评审-优先级联动机制**

当评审Agent输出优先级调整建议时：

1. 将建议解析为结构化的 `{test_point: suggested_priority}` 映射
2. 在用例生成完成后，自动应用优先级调整
3. 如果调整后的优先级与LLM生成的优先级不一致，以评审建议为准（评审视角更贴近需求本质）
