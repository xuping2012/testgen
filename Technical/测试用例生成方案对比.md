## 4. 方案A: 当前方案（一次性批量生成）

### 4.1 架构

```
[需求文档] → Phase 1: 测试计划 → 用户确认
    ↓
Phase 2: 
    ├─ 构建完整Prompt (包含全部ITEM/POINT)
    ├─ RAG召回 (Top 10)
    ├─ LLM一次性生成 (max_tokens=8192, timeout=180)
    ├─ 解析JSON
    └─ 批量落库
```

### 4.2 代码实现

```python
# src/services/generation_service.py
def execute_phase2(self, task_id, requirement, test_plan):
    # 1. RAG召回
    rag_context = self.vector_store.search(
        query=requirement.content,
        top_k=10
    )
    
    # 2. 构建Prompt
    prompt = self.build_prompt(
        test_plan=test_plan,
        rag_context=rag_context,
        requirement=requirement
    )
    
    # 3. LLM调用
    response = adapter.generate(
        prompt,
        max_tokens=8192,
        timeout=180,
        max_retries=3
    )
    
    # 4. 解析
    test_cases = self._parse_generated_cases(response.content)
    
    # 5. 落库
    self._save_cases_to_db(test_cases, requirement.id)
```

### 4.3 优劣势

**优势**：

- ✅ 实现简单，代码量少
- ✅ 上下文完整（一次传入全部需求）
- ✅ 用户只需确认一次（Phase 1）

**劣势**：

- ❌ **max\_tokens限制**：最多40条用例
- ❌ **超时风险高**：180秒经常不够
- ❌ **JSON解析失败率高**：长格式容易出错
- ❌ **失败成本高**：失败后全部重来
- ❌ **用例质量不稳定**：LLM长输出质量下降
- ❌ **无进度反馈**：用户只能等待

### 4.4 适用场景

- 需求文档简单（<10个测试点）
- 用例数量少（<30条）
- 快速原型验证

    └─ ...

### 5.2 核心设计

#### 5.2.1 全局上下文准备（只执行一次）

```python
def prepare_generation_context(self, requirement, test_plan):
    """准备生成上下文（Phase 2开始时执行一次）"""
    return {
        # 测试计划摘要（不是完整JSON）
        "plan_summary": {
            "total_modules": len(test_plan.items),
            "total_points": sum(len(item.points) for item in test_plan.items),
            "core_modules": [item.title for item in test_plan.items if item.priority == 'P0'],
            "global_constraints": self.extract_constraints(test_plan)
        },
        
        # 业务规则提取
        "business_rules": self.extract_business_rules(test_plan),
        # 例如：
        # [
        #   "订单金额>1000需要审批",
        #   "支付超时30分钟自动取消",
        #   "用户必须实名认证"
        # ]
        
        # RAG召回（按模块索引）
        "rag_context": {
            "global_examples": self.rag_search(requirement.content, top_k=5),
            "module_examples": {
                item.id: self.rag_search(item.title, top_k=3)
                for item in test_plan.items
            }
        },
        
        # 生成策略
        "generation_strategy": {
            "priority_rules": "核心功能P0，辅助功能P1，边界条件P2",
            "naming_convention": "[模块]-[场景]-[类型]",
            "coverage_rule": "每个POINT至少2条用例（正向+逆向）",
            "style_guide": "用例格式：[前置条件][操作步骤][预期结果][优先级]"
        }
    }
```

#### 5.2.2 分批生成（每个ITEM独立）

```python
def generate_item_cases(self, item, global_context, recent_cases):
    """为单个ITEM生成用例"""
    prompt = f"""
你是测试专家，请为以下功能模块生成测试用例。

===== 全局上下文 =====
【测试计划摘要】
- 总模块数: {global_context['plan_summary']['total_modules']}
- 总测试点: {global_context['plan_summary']['total_points']}
- 核心模块: {', '.join(global_context['plan_summary']['core_modules'])}

【业务规则】（必须遵守）
{chr(10).join(f'- {rule}' for rule in global_context['business_rules'])}

【参考用例】（RAG召回）
{self.format_rag_examples(global_context['rag_context']['module_examples'].get(item.id, []))}

【生成策略】
- 优先级规则: {global_context['generation_strategy']['priority_rules']}
- 命名规范: {global_context['generation_strategy']['naming_convention']}
- 覆盖要求: {global_context['generation_strategy']['coverage_rule']}
- 格式要求: {global_context['generation_strategy']['style_guide']}

===== 最近生成的用例 =====（保持风格一致）
{self.format_recent_cases(recent_cases)}

===== 当前任务 =====
【模块名称】: {item.title}
【测试点列表】:
{chr(10).join(f'- {point}' for point in item.points)}

请为以上测试点生成 {len(item.points) * 2} 条测试用例。

输出格式（JSON数组）:
[
  {{
    "case_title": "用例标题",
    "test_steps": ["步骤1", "步骤2", "步骤3"],
    "expected_result": "预期结果",
    "priority": "P0/P1/P2"
  }}
]
"""
    
    response = adapter.generate(
        prompt,
        temperature=0.7,
        max_tokens=4096,  # 每个ITEM只需20条
        timeout=120       # 单项超时
    )
    
    return self._parse_generated_cases(response.content)
```

#### 5.2.3 分批执行逻辑

```python
def execute_phase2_batch(self, task_id, requirement, test_plan):
    """分批生成用例"""
    # 1. 准备全局上下文
    global_context = self.prepare_generation_context(requirement, test_plan)
    
    # 2. 分批生成
    all_cases = []
    total_items = len(test_plan.items)
    
    for idx, item in enumerate(test_plan.items, 1):
        # 更新进度
        progress = 30 + (idx / total_items) * 60
        self.update_progress(
            task_id, 
            progress, 
            f"正在生成模块 {idx}/{total_items}: {item.title}"
        )
        
        # 获取最近5条用例（保持连贯）
        recent_cases = all_cases[-5:]
        
        try:
            # 生成当前ITEM的用例
            cases = self.generate_item_cases(item, global_context, recent_cases)
            
            # 立即落库
            saved = self._save_cases_to_db(cases, requirement.id)
            all_cases.extend(saved)
            
        except Exception as e:
            # 单个ITEM失败，记录日志继续下一个
            self.log_error(f"ITEM-{idx} [{item.title}] 生成失败: {e}")
            continue
    
    # 3. 完成
    self.complete_task(task_id, stats={
        "total_cases": len(all_cases),
        "success_items": len(all_cases),
        "failed_items": total_items - len(all_cases)
    })
```

### 5.3 优劣势

**优势**：

- ✅ **解决超时问题**：每个ITEM独立120秒，互不影响
- ✅ **支持任意数量用例**：100个ITEM × 20条 = 2000条
- ✅ **用例质量好**：短输出质量远高于长输出
- ✅ **失败可重试**：单个ITEM失败不影响其他
- ✅ **实时进度**：用户看到"正在生成模块3/10"
- ✅ **Token消耗合理**：比方案A高50%，但成功率高
- ✅ **保持连贯性**：共享上下文 + 最近用例

**劣势**：

- ⚠️ **生成耗时增加**：从15分钟增加到12分钟（但成功率更高）
- ⚠️ **代码复杂度增加**：需要管理全局上下文
- ⚠️ **仍然无质检**：重复用例和覆盖度无法保证

### 5.4 适用场景

- 需求文档复杂（10-50个测试点）
- 用例数量中等（50-500条）
- 对质量有一定要求但不苛刻

#### 6.2.1 重复检测

```python
def detect_duplicates(self, cases):
    """检测重复用例（语义相似度）"""
    from sklearn.feature_extraction.text import TfidfVectorizer
    from sklearn.metrics.pairwise import cosine_similarity
    
    # 提取用例描述
    descriptions = [
        f"{c['case_title']} {' '.join(c['test_steps'])}"
        for c in cases
    ]
    
    # TF-IDF向量化
    vectorizer = TfidfVectorizer()
    vectors = vectorizer.fit_transform(descriptions)
    
    # 计算相似度矩阵
    similarity_matrix = cosine_similarity(vectors)
    
    # 标记重复（相似度>85%）
    duplicates = []
    for i in range(len(cases)):
        for j in range(i+1, len(cases)):
            if similarity_matrix[i][j] > 0.85:
                duplicates.append({
                    "case1": cases[i],
                    "case2": cases[j],
                    "similarity": similarity_matrix[i][j]
                })
    
    return duplicates

def remove_duplicates(self, cases, duplicates):
    """去重：保留优先级高的"""
    to_remove = set()
    for dup in duplicates:
        # 保留P0，删除P1/P2
        if dup['case1']['priority'] < dup['case2']['priority']:
            to_remove.add(id(dup['case2']))
        else:
            to_remove.add(id(dup['case1']))
    
    return [c for c in cases if id(c) not in to_remove]
```

#### 6.2.2 覆盖度检查

```python
def check_coverage(self, test_plan, cases):
    """检查测试点覆盖度"""
    # 构建测试点→用例映射
    coverage_map = {}
    for case in cases:
        # 从用例标题或标签提取测试点
        point_id = self.extract_point_id(case['case_title'])
        if point_id:
            if point_id not in coverage_map:
                coverage_map[point_id] = []
            coverage_map[point_id].append(case)
    
    # 计算覆盖度
    total_points = sum(len(item.points) for item in test_plan.items)
    covered_points = len(coverage_map)
    coverage_rate = covered_points / total_points
    
    # 找出未覆盖的测试点
    uncovered_points = []
    for item in test_plan.items:
        for point in item.points:
            if point.id not in coverage_map:
                uncovered_points.append({
                    "item_id": item.id,
                    "point_id": point.id,
                    "point_title": point.title
                })
    
    return {
        "total_points": total_points,
        "covered_points": covered_points,
        "coverage_rate": coverage_rate,
        "uncovered_points": uncovered_points
    }
```

#### 6.2.3 质量评分

```python
def quality_score(self, cases):
    """用例质量评分"""
    scores = []
    
    for case in cases:
        score = 0
        
        # 格式完整性（30分）
        if case.get('case_title'): score += 10
        if case.get('test_steps') and len(case['test_steps']) > 0: score += 10
        if case.get('expected_result'): score += 10
        
        # 步骤合理性（30分）
        step_count = len(case.get('test_steps', []))
        if 2 <= step_count <= 10: score += 20  # 步骤数量合理
        if all(len(s) > 5 for s in case.get('test_steps', [])): score += 10  # 步骤描述详细
        
        # 优先级合理性（20分）
        if case.get('priority') in ['P0', 'P1', 'P2', 'P3']: score += 20
        
        # 边界条件覆盖（20分）
        content = f"{case['case_title']} {' '.join(case['test_steps'])}"
        if any(kw in content for kw in ['异常', '边界', '超时', '失败', '错误']):
            score += 20
        
        scores.append(score)
    
    return {
        "average_score": sum(scores) / len(scores),
        "min_score": min(scores),
        "max_score": max(scores),
        "high_quality_count": len([s for s in scores if s >= 80]),
        "low_quality_count": len([s for s in scores if s < 60])
    }
```

def generate_missing_cases(self, uncovered_points, global_context):
    """为未覆盖的测试点补充生成用例"""
   补充_cases = []
    
    for point in uncovered_points:
        prompt = f"""

【全局上下文】
{global_context['plan_summary']}

【测试点信息】
模块: {point['item_id']}
测试点: {point['point_title']}

请为该测试点生成2条用例（正向+逆向）。
"""
        
        response = adapter.generate(prompt, max_tokens=2048, timeout=60)
        cases = self._parse_generated_cases(response.content)
        补充_cases.extend(cases)
    
    return 补充_cases

## 7. 方案D: 分批并行生成 + 后置质检

### 7.1 架构

```

    ↓

    ├─ Thread-2: ITEM-2, ITEM-5, ITEM-8 → LLM → 落库
    └─ Thread-3: ITEM-3, ITEM-6, ITEM-9 → LLM → 落库

    ├─ 重复检测
    ├─ 覆盖度检查
    └─ 质量评分
         ↓
Phase 4 (补充生成):
    └─ 覆盖度<90%的测试点重新生成
         ↓
    └─ 最终落库 + 完成
```

### 7.2 核心设计

#### 7.2.1 并行生成

```python
from concurrent.futures import ThreadPoolExecutor, as_completed

def execute_phase2_parallel(self, task_id, requirement, test_plan):
    """并行分批生成用例"""
    # 1. 准备全局上下文
    global_context = self.prepare_generation_context(requirement, test_plan)
    
    # 2. 并行生成（最多3个线程）
    all_cases = []
    max_workers = min(3, len(test_plan.items))  # 最多3个并发
    
    with ThreadPoolExecutor(max_workers=max_workers) as executor:
        # 提交任务
        future_to_item = {}
        for idx, item in enumerate(test_plan.items):
            future = executor.submit(
                self.generate_item_cases_safe,
                item=item,
                global_context=global_context,
                recent_cases=[],  # 并行时无最近用例
                task_id=task_id,
                idx=idx
            )
            future_to_item[future] = item
        
        # 收集结果
        for future in as_completed(future_to_item):
            item = future_to_item[future]
            try:
                cases = future.result()
                all_cases.extend(cases)
            except Exception as e:
                self.log_error(f"ITEM [{item.title}] 生成失败: {e}")
    
    # 3. 后置质检
    quality_report = self.run_quality_check(all_cases, test_plan)
    
    # 4. 补充生成（如有必要）
    if quality_report['coverage_rate'] < 0.9:
        补充_cases = self.generate_missing_cases(
            quality_report['uncovered_points'],
            global_context
        )
        all_cases.extend(补充_cases)
    
    # 5. 完成
    self.complete_task(task_id, stats={
        "total_cases": len(all_cases),
        "quality_report": quality_report
    })
```

### 7.3 优劣势

**优势**：

- ✅ **生成速度快**：3个并发，耗时降低60%
- ✅ **质量好**：后置质检保证
- ✅ **适合大规模**：1000+用例也能快速生成

**劣势**：

- ❌ **上下文连贯性差**：并行时无法共享最近用例
- ❌ **实现复杂度高**：需要处理并发、锁、线程安全
- ❌ **LLM调用成本高**：并发调用可能触发API限流
- ❌ **调试困难**：并发问题难以复现

### 7.4 适用场景

- 超大规模需求（100+测试点）
- 对生成速度有高要求
- 有充足的LLM调用配额

## 8. 人工审核可行性分析

### 8.1 方案对比

| 方案                     | 人工介入点           | 审核次数          | 总耗时       | 可行性   |
| ---------------------- | --------------- | ------------- | --------- | ----- |
| **A. 事中审核（每5个ITEM暂停）** | 生成过程中           | 20次（100个ITEM） | 100-200分钟 | ❌ 不可行 |
| **B. 事前策略 + 事后审核**     | 生成前配置策略，生成后审核结果 | 2次            | 15-20分钟   | ✅ 可行  |
| **C. 事后审核 + 质量报告**     | 生成后审核质检报告       | 1次            | 5-10分钟    | ✅ 可行  |

### 8.2 "每5个ITEM暂停审核"的问题

**❌ 违背自动化初衷**：

```
目标: 减少人工，提升效率
现实: 需要人守着电脑，比人工编写还慢
```

**❌ 打断生成流程**：

```
问题1: LLM异步调用，暂停再恢复增加复杂度
问题2: 用户可能离开电脑，导致生成中断
问题3: 上下文需要序列化存储，增加成本
```

**❌ 不切实际**：

```
100个ITEM = 20次审核
每次审核5-10分钟 = 100-200分钟
比人工编写用例还慢！
```

### 8.3 正确的人工介入点

#### ✅ 事前：策略配置（1次，2-3分钟）

```
用户在Phase 1确认测试计划时，可以配置：
- 生成策略：激进（每个POINT 3条）/ 标准（2条）/ 精简（1条）
- 优先级分配：核心模块P0 / 辅助模块P1
- 质量要求：覆盖度阈值（80%/90%/95%）
```

**代码实现**：

```python
# 前端：Phase 1确认时弹窗
{

    "coverage_rule": "standard",  // aggressive/standard/minimal
    "priority_allocation": "core_p0",  // core_p0/all_p0
    "quality_threshold": 0.9  // 0.8/0.9/0.95
  }
}
```

#### ✅ 事后：质量报告审核（1次，3-5分钟）

```
生成完成后，展示质检报告：
- 总用例数: 185条
- 覆盖度: 94% (未覆盖6个测试点)
- 重复率: 5% (已去重12条)
- 质量评分: 8.2/10
- 低质量用例: 8条（<60分）

用户操作:
[确认入库] [补充生成] [查看详情] [重新生成]
```

**代码实现**：

### 8.4 结论

**❌ 不推荐**: 事中审核（每5个ITEM暂停）

**✅ 推荐**: 事前策略配置 + 事后质量报告审核

**理由**:

1. 符合自动化目标（减少人工）
2. 用户只需介入2次（事前1次，事后1次）
3. 总耗时<10分钟
4. 质量可控（有质检报告支撑）
